# DDR GP Score Tracker デザインシステム

## 1. 目的

本書は、DDR GP Score Tracker の画面全体で使用する視覚表現、コンポーネント、色、余白、状態表現を定義する。

画面サンプル画像は視覚参考とする。実装時は本書のルールを優先し、画像のピクセル完全一致を目的としない。

画像内に本書と矛盾する配色、重複要素、誤字、データ不整合がある場合は、本書を優先する。

---

## 2. デザイン原則

### 2.1 一般ユーザー優先

開発、OCR、画像解析の知識を前提にしない。

通常画面で避ける表現：

- ROI
- OCR確信度
- キャプチャ方式
- マスタDB
- 解析失敗
- プロセス監視
- ウィンドウハンドル
- 画像ハッシュ

ユーザー向けには、結果、状態、次の行動を表示する。

### 2.2 スコアが主役

画面の視覚的優先順位は以下とする。

1. 曲名
2. 通常スコア
3. EXスコア・EXスコア率
4. プレイスタイル・難易度・レベル
5. 自己ベスト差
6. 判定数
7. 自動記録状態

自動記録状態や技術情報は補助的に扱う。

### 2.3 情報を色だけで伝えない

色に加えて、文字、符号、アイコン、ラベルを併用する。

例：

- `↑ +12,340`
- `↓ -2,180`
- `＝ ±0`
- `初プレー`

### 2.4 ジャケット画像を使用しない

楽曲ジャケット画像は使用しない。

カードは曲名、プレイスタイル、難易度、レベル、スコア情報だけで成立させる。

### 2.5 仕様上の状態を重複表示しない

同じ状態情報を、ヘッダーとサイドバーの両方に重複表示しない。

自動記録状態は、サイドバー下部にのみ表示する。

### 2.6 データ整合性を視覚表現より優先する

見栄えのために、現在の選択条件と矛盾するデータを表示してはならない。

例：

- SINGLE選択時にDP譜面を表示しない
- DOUBLE選択時にSP譜面を表示しない
- 適用中フィルターと矛盾する行を表示しない

## 4. 難易度カラー

DDRプレイヤーが直感的に識別できる色を使用する。ただし、公式画面の完全再現ではなく、可読性を優先して調整する。

| 難易度 | 基本色 | 推奨背景 | 推奨文字 |
|---|---|---|---|
| BEGINNER | 水色 | #5FB8E8 | #0B2330 |
| BASIC | 黄〜オレンジ | #F3A621 | #2B1A00 |
| DIFFICULT | 赤 | #E64949 | #FFFFFF |
| EXPERT | 緑 | #27A85A | #FFFFFF |
| CHALLENGE | 紫 | #8C55C9 | #FFFFFF |

色だけではなく、必ず難易度名または略称を表示する。

### 4.1 バッジ表記

通常：

- BEGINNER
- BASIC
- DIFFICULT
- EXPERT
- CHALLENGE

横幅不足時：

- BGN
- BAS
- DIF
- EXP
- CHA

### 4.2 プレイスタイル表記

- SP
- DP

プレイスタイルと難易度は、別のバッジまたは別列として表示する。

### 4.3 レベル表記

- カード：`Lv.17`
- テーブル：数値列 `17` を許容
- プレイスタイルや難易度に統合しない

---

## 5. タイポグラフィ

### 5.1 推奨フォント

Windows標準環境での可読性を優先する。

- Segoe UI Variable
- Segoe UI
- Yu Gothic UI

日本語表示はOSのフォールバックを許容する。

### 5.2 サイズ

| 用途 | サイズ | ウェイト |
|---|---:|---:|
| 画面タイトル | 24px | 600 |
| セクションタイトル | 18px | 600 |
| カードタイトル | 14px | 600 |
| 通常スコア | 28〜36px | 700 |
| EXスコア | 18〜24px | 700 |
| KPI数値 | 24〜30px | 600 |
| 本文 | 14px | 400 |
| 補助情報 | 12px | 400 |
| テーブル見出し | 12px | 600 |
| バッジ | 11〜12px | 600 |

通常スコアを主、EXスコアを第二指標として配置する。ただし、EXスコア率は明確に読める大きさを維持する。

### 5.3 数値

スコア、判定数、件数は、可能なら等幅数字を使用する。

- 通常スコア：`987,650`
- EXスコア：`2,596 / 2,690`
- EXスコア率：`96.5%`
- 自己ベスト差：符号を必ず表示
- パーセントは小数1桁を基本とする

---

## 6. レイアウト

### 6.1 基本グリッド

8px単位を基本とする。

推奨余白：

- 画面外周：24px
- セクション間：24px
- カード間：16px
- カード内余白：16px
- ラベルと値：4〜8px

### 6.2 サイドバー

- 幅：160〜200px
- 背景：BackgroundSidebar
- 選択中項目はAccentPrimary背景と白文字
- 上部にアプリ名と主要ナビゲーション
- 下部に設定、データ管理、自動記録状態
- 自動記録状態をヘッダーへ重複表示しない

### 6.3 ヘッダー

ヘッダーには以下を配置できる。

- 画面タイトル
- 通知
- テーマ切り替え
- その他メニュー

自動記録状態は配置しない。

### 6.4 メイン領域

- 最大幅を固定しすぎない
- 1200×760程度で主要画面が成立すること
- 最小サイズは960×640を目安とする
- 横幅不足時はテーブル列を段階的に省略する
- 主要操作を横スクロール前提にしない

### 6.5 ホーム下段

`最近の自己ベスト更新` を横幅いっぱい、または十分な幅で表示する。

仕様にない `自己ベスト更新ランキング（今日）` を追加しない。

---

## 7. コンポーネント

### 7.1 カード

用途：

- 今日のプレー状況
- 最新プレー
- 分類別サマリー
- 最近の自己ベスト更新

仕様：

- 角丸：8〜12px
- 枠線：1px
- 影は控えめ
- カード内に過度な装飾を置かない
- ジャケット画像用の空き領域を作らない

### 7.2 KPIカード

構成：

- 大きな数値
- 短いラベル
- 必要な場合のみ補足

例：

```text
12
プレー数
```

平均通常スコアはKPIに使用しない。

### 7.3 難易度バッジ

- 高さ：20〜24px
- 左右余白：8px
- 角丸：4〜6px
- 文字は中央揃え
- 背景色と文字色はテーマごとに確認する

### 7.4 テーブル

- 行高：36〜44px
- 見出しは固定可能にする
- 行ホバー状態を持つ
- 行選択で詳細画面または詳細ペインを開く
- 数値列は右揃え
- 曲名は左揃え
- プレイスタイルと難易度は別列
- 現在の分類やフィルターと矛盾する行を表示しない

### 7.5 タブ

使用箇所：

- SINGLE / DOUBLE
- レベル別 / バージョン別 / タイトル別
- 通常スコア推移 / EXスコア率推移 / 判定推移 / プレー履歴

最上位切り替えと下位タブは、視覚的階層を変える。

- SINGLE / DOUBLE：セグメントコントロール
- 分類切り替え：タブ

### 7.6 フィルター

使用コンポーネント：

- コンボボックス
- 検索ボックス
- 日付範囲選択
- 詳細フィルターボタン
- 適用中条件チップ

フィルター数が多い場合は、以下の構成とする。

- 使用頻度の高い条件：常時表示
- 使用頻度の低い条件：詳細フィルター
- 適用中条件：チップ表示

1行に詰め込みすぎず、2行表示または折りたたみを許容する。

### 7.7 フィルターチップ

例：

```text
[Lv.17 ×]
[AAA未達成 ×]
[EX率95%未満 ×]
```

- 個別解除を可能にする
- 全解除操作を用意する
- 色だけで適用状態を示さない

### 7.8 空状態

空状態では以下を表示する。

- 状況を示す短い見出し
- 次の行動を示す1文
- 必要な場合のみアクションボタン

技術的な原因説明は表示しない。

---

## 8. スコア表現

### 8.1 通常スコア

- 大きく表示する
- ランクを隣接表示する
- 自己ベスト差を明示する

### 8.2 EXスコア

表示形式：

```text
EX SCORE
2,596 / 2,690   96.5%
```

一覧では以下の省略を許容する。

```text
2,596 / 96.5%
```

横幅がさらに不足する場合は、EXスコア率を優先する。

### 8.3 更新状態

通常スコア更新とEXスコア更新を分ける。

統一ルール：

- 更新：AccentPrimary
- 未更新：Danger
- タイ：TextMuted
- 初プレー：Info

表示例：

- 更新：`↑ +12,340`
- 未更新：`↓ -2,180`
- タイ：`＝ ±0`
- 初プレー：`初プレー`

同じ状態に、画面やコンポーネントごとに異なる色を使用しない。

赤色をプラス更新に使用しない。

色だけでなく、矢印、符号、ラベルを併用する。

---

## 9. 判定表示

表示順：

1. MARVELOUS
2. PERFECT
3. GREAT
4. GOOD
5. O.K.
6. MISS

最大コンボは判定とは分けて表示する。

`N.G.` は表示しない。

ホームのカード高さを抑える場合は2列配置を許容するが、論理順は維持する。

---

## 10. グラフ

### 10.1 共通

- 1つのグラフに過度な系列を重ねない
- ホバーまたは選択で値を確認できる
- 点と線だけでなく、軸ラベルを表示する
- ライト・ダークテーマ双方で補助線が読めること
- 選択中の表示軸を明示する

### 10.2 通常スコア推移

- 線グラフ
- 自己ベスト更新点を識別可能にする
- ランク境界線は補助的に表示する
- 全プレー / 自己ベスト推移を切り替えられる

### 10.3 EXスコア率推移

- 縦軸は百分率
- 値は小数1桁を基本とする
- 必要に応じて95%などの補助線を表示する
- 全プレー / 自己ベスト推移を切り替えられる

### 10.4 判定推移

- 1指標ずつ表示する
- MISS数など、減少が改善となる指標は色だけで良否を決めない

---

## 11. アイコン

- Windows標準またはFluent系アイコンを使用する
- 曲名右側の星アイコンは使用しない
- アイコン単体で意味を伝えず、必要に応じてラベルを併記する
- 通知、テーマ、メニュー等の一般的な用途に限定する

---

## 12. 文言

### 12.1 推奨表現

- 準備できています
- DDR GRAND PRIXを待っています
- プレー結果を保存しました
- このプレーを確認してください
- まだプレーデータがありません
- 自動記録：待機中
- 自動記録：準備完了

### 12.2 避ける表現

- OCR失敗
- 解析エラー
- ROI設定
- ウィンドウキャプチャ失敗
- マスタDB不一致
- プロセス監視中

技術情報はサポート情報に限定する。

---

## 13. アクセシビリティ

- キーボード操作に対応する
- フォーカス状態を明確にする
- 色だけで状態を表現しない
- 文字サイズを極端に小さくしない
- ライト・ダーク双方でコントラストを確認する
- バッジには必ず文字を含める
- テーブル行はキーボードで選択できること
- 詳細フィルターはキーボードで開閉できること

---

## 14. Codex実装時の優先順位

1. `screen-spec.md` を優先する
2. 本デザインシステムを優先する
3. 画面サンプル画像は配置と視覚的優先順位の参考にする
4. 画像内の誤字、生成上の不整合、仕様にない要素を実装しない
5. ピクセル完全一致より、再利用可能なコンポーネント構成を優先する
6. 色はハードコードせず、テーマトークンを使用する
7. プレイスタイル、難易度、レベルを別属性として扱う
8. SINGLE / DOUBLE と異なるプレイスタイルのデータを混在させない
9. 更新状態の色と符号を全画面で統一する
10. 自動記録状態を複数箇所へ重複表示しない
