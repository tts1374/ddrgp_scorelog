# 管理・設定 / 収集状況 改善ワイヤーフレーム

manual review運用完成後の後続Goal候補として、管理・設定画面と収集状況画面の改善案を保存する。

この文書は現時点の画面仕様候補であり、`docs/next-task.md`へ昇格するまでは実装対象としない。

## 目的

- UI上の`master` / `catalog`という内部用語を、利用目的が分かる表現へ置き換える。
- 利用者にDBファイルを手動選択させず、アプリ自身が曲情報とジャケット情報を初期化・更新する。
- 曲情報、ジャケット情報、収集状況の現在状態を一目で確認できるようにする。
- 収集状況画面の内部statusを日本語ラベルへ置き換える。

## 全体用語

| 現在の表示 | 変更後 |
|---|---|
| master | 曲情報 |
| catalog | ジャケット情報 |
| master hash | 曲情報の更新日時 |
| catalog version | ジャケット情報のversion |

画面右上の表示も、内部hashではなく以下のような利用者向け情報へ変更する。

```text
曲情報: 2026/07/19 10:00 更新
ジャケット情報: v1
```

`master` / `catalog`の手動選択UIは削除する。利用者は1人であり、異常時はDBを直接操作できるため、詳細設定や開発者向けUIへの退避も行わない。

# 管理・設定

## 基本動作

- 起動時に既定保存場所から曲情報DBとジャケット情報DBを読み込む。
- 曲情報DBが存在しない場合は、未作成状態として表示する。
- ジャケット情報DBが存在しない場合は、利用可能な曲情報を基準として自動作成する。
- `曲情報を更新`で曲情報DBを更新する。
- DB読み込み中、曲情報更新中、ジャケット情報初期作成中、収集中は`曲情報を更新`を押せない。
- 初期化・更新処理中も、利用可能な既存情報は可能な範囲で表示する。
- 操作ボタンは`曲情報を更新`だけとし、`処理を取消`は削除する。

## 曲情報がない場合

```text
曲情報                         ジャケット情報
曲情報がありません。           読み込み中…

[ 曲情報を更新 ]
```

## DB読み込み中

```text
曲情報                         ジャケット情報
読み込み中…                    読み込み中…

[ 曲情報を更新 ]  ※無効
```

## ジャケット情報初期作成中

```text
曲情報                         ジャケット情報
更新日時: 2026/07/19 10:00:00  初期情報を作成中…
曲: 1181曲
譜面: 9594譜面

[ 曲情報を更新 ]  ※無効
```

## 曲情報更新中

```text
曲情報                         ジャケット情報
曲情報を更新しています…        version: v1
                               収集状況: 120 / 1181
                               レビュー中: 10件

[ 曲情報を更新 ]  ※無効
```

## 収集中

```text
曲情報                         ジャケット情報
更新日時: 2026/07/19 10:00:00  version: v1
曲: 1181曲                     収集状況: 120 / 1181
譜面: 9594譜面                 レビュー中: 10件

[ 曲情報を更新 ]  ※無効
```

## 通常状態

```text
曲情報                         ジャケット情報
更新日時: 2026/07/19 10:00:00  version: v1
曲: 1181曲                     収集状況: 120 / 1181
譜面: 9594譜面                 レビュー中: 10件

[ 曲情報を更新 ]
```

## 更新失敗時

既存DBが利用可能なら、更新失敗によって現在の情報を失わない。

```text
曲情報                         ジャケット情報
更新に失敗しました。            version: v1
前回更新: 2026/07/19 10:00:00   収集状況: 120 / 1181
曲: 1181曲                     レビュー中: 10件
譜面: 9594譜面

[ 曲情報を更新 ]

エラー: <短い理由>
```

詳細なstack traceは通常画面へ表示しない。

更新は一時ファイルまたは一時DBへ作成し、検証成功後に既存DBと置換する。更新失敗時は既存DBをそのまま維持する。

## 処理の競合

曲情報またはジャケット情報に対する初期化・更新処理中は、同種の処理を重ねて開始しない。

収集中は`曲情報を更新`を無効化する。収集を自動停止して更新を開始する動作は行わない。

# ジャケット情報の初期作成と曲情報更新への追従

曲情報の読み込みに成功し、ジャケット情報DBが存在しない場合に自動作成する。

最低限、以下を作成する。

- catalog schema
- catalog metadata
- Master上のGP対象曲を未収集として扱える初期状態

全曲分のrowを事前作成するか、曲情報とのjoinで未収集を導出するかは、既存schemaと実装を確認して決める。画面上の結果は同じにする。

- GP対象曲が収集状況へ表示される。
- 初期状態は未収集になる。
- orphanや未割当を生成しない。

曲情報更新時は以下のように追従する。

- 新しく追加されたGP対象曲を未収集として収集状況へ追加する。
- 既存のジャケット情報、reference、review結果を失わない。
- GP対象外化または曲情報上の削除が発生しても、既存ジャケット情報を物理削除しない。
- GP対象外化または曲情報上の削除によって通常の収集対象から外れても、保存済み情報は残す。
- 曲情報に存在しなくなった保存済みデータは、必要に応じてorphanとして診断可能にする。

# 収集状況

## 集計表示

現在の内部用語を日本語へ変更する。

```text
収集済み: 120 / レビュー待ち: 10 / 未収集: 1051 / 曲未特定: 0
曲情報に存在しないデータ: 0 / 未割当: 0
```

管理・設定画面の`収集状況: 120 / 1181`は、以下の定義とする。

- 分子: 収集済み曲数
- 分母: 現在のGP対象曲数
- レビュー待ちは分子へ含めない

`レビュー中`は、レビュー画面の`未レビュー`タブに表示されている件数とする。

- 未レビュー
- 保留
- 画面上またはODS importで確定予定・reject予定になっているが未反映のもの

確定済み・reject反映済みは含めない。

## 状態ラベル

| 内部値 | 表示 |
|---|---|
| `all` | すべて |
| `referenced` | 収集済み |
| `needs_review` | レビュー待ち |
| `uncollected` | 未収集 |
| `unresolved` | 曲未特定 |
| `orphan` | 曲情報に存在しないデータ |

`未割当 unresolved`のように内部分類を連結した表示は避け、人間向けの独立したラベルを定義する。

## 理由ラベル

理由プルダウンと理由列も、内部値をそのまま表示しない。

実装時に実在するreason値を列挙し、以下の形式で表示マッピングを定義する。

| 内部値 | 表示例 |
|---|---|
| capture mismatch系 | 取得画面が一致しない |
| OCR unresolved系 | 曲名を特定できない |
| duplicate系 | 重複候補 |
| missing image系 | 画像が見つからない |
| stale / drift系 | 元データが更新されている |

未知のreason値は隠さず、`不明: <内部値>`として表示する。

## 一覧列

基本列は以下とする。

```text
状態 / 曲名 / アーティスト / 登録数 / 理由 / song ID
```

- `song ID`は診断に有用なため残してよい。
- `登録数`が何を数えるかは、既存実装を確認して表示名を具体化する。
  - ジャケット画像数なら`登録画像数`
  - reference数なら`登録数`
- 状態・理由は日本語表示にする。

# 実装前にコードから確認する事項

仕様判断として未確定なのではなく、実装方法を決めるために既存コードとschemaを確認する。

1. **更新日時の保存場所**
   - DBファイルのmtimeではなく、曲情報の正常更新完了日時を表示する。
   - 既存metadataに時刻がなければ、最小限の保存方法を追加する。

2. **曲情報の更新元**
   - 現在の更新元、取得処理、入力検証を確認する。
   - ネットワーク失敗または入力不正時は既存DBを維持する。

3. **新曲追加の追従方法**
   - catalogへ初期rowを追加する方式か、Masterとのjoinで未収集を導出する方式かを既存schemaに合わせる。

4. **version表示**
   - 既存schemaに保存されている値を事実どおり表示する。
   - 整数versionが`1`なら`v1`と表示し、UIだけで`1.0`へ変換しない。

5. **登録数の実体**
   - 現在の列が画像数、reference数、observation数のどれを数えているか確認し、表示名を確定する。

# 非要件

- 任意のmaster/catalogファイルを選択する通常UI
- master/catalog選択機能の詳細設定または開発者向けUIへの退避
- 詳細なDB管理画面
- DB hashの通常表示
- migration履歴viewer
- stack traceの通常表示
- 複数catalogの切り替え
- 一般ユーザー向けの保存場所カスタマイズ
- 処理のキャンセルUI
- GP対象外化または削除曲に紐づく既存ジャケット情報の物理削除

# Goal候補の完了条件

- UI上の`master` / `catalog`表記が`曲情報` / `ジャケット情報`へ置き換わる。
- hashではなく曲情報の正常更新日時を表示する。
- DBファイルを手動選択せず起動・初期化できる。
- 曲情報がない状態から画面操作で更新できる。
- ジャケット情報DBがない場合に自動作成される。
- 読み込み・更新・初期作成・収集中・通常・失敗状態が区別して表示される。
- DB処理中および収集中は更新ボタンが無効になる。
- 操作ボタンが`曲情報を更新`だけになる。
- 収集状況と理由が日本語ラベルで表示される。
- 収集状況の分子が収集済み曲数として表示される。
- レビュー中件数が未レビュータブの表示件数と一致する。
- 新曲追加後も既存ジャケット情報を失わず、未収集曲として追従する。
- GP対象外化または削除された曲の既存ジャケット情報を物理削除しない。
- 更新失敗時に既存DBが維持される。
- 対象test、Ruff、構文検査、`git diff --check`が成功する。
