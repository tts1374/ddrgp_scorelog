# 管理・設定 / 収集状況 改善ワイヤーフレーム

manual review運用完成後の後続Goal候補として、管理・設定画面と収集状況画面の改善案を保存する。

この文書は現時点の画面仕様候補であり、`docs/next-task.md`へ昇格するまでは実装対象としない。

## 目的

- UI上の`master` / `catalog`という内部用語を、利用目的が分かる表現へ置き換える。
- 利用者にDBファイルを手動選択させず、アプリ自身が曲情報とジャケット情報を初期化・更新する。
- 曲情報、ジャケット情報、収集状況の現在状態を一目で確認できるようにする。
- 収集状況画面の内部statusを日本語ラベルへ置き換える。

## 全体用語

| 現在の表示 | 変更後 |
|---|---|
| master | 曲情報 |
| catalog | ジャケット情報 |
| master hash | 曲情報の更新日時 |
| catalog version | ジャケット情報のversion |

画面右上の表示も、内部hashではなく以下のような利用者向け情報へ変更する。

```text
曲情報: 2026/07/19 10:00 更新
ジャケット情報: v1.0
```

`master` / `catalog`の手動選択UIは削除する。利用者は1人であり、異常時はDBを直接操作できるため、詳細設定や開発者向けUIへの退避も必須としない。

# 管理・設定

## 基本動作

- 起動時に既定保存場所から曲情報DBとジャケット情報DBを読み込む。
- 曲情報DBが存在しない場合は、未作成状態として表示する。
- ジャケット情報DBが存在しない場合は、利用可能な曲情報を基準として自動作成する。
- `曲情報を更新`で曲情報DBを更新する。
- DB読み込み中、曲情報更新中、ジャケット情報初期作成中は`曲情報を更新`を押せない。
- 初期化・更新処理中も、利用可能な既存情報は可能な範囲で表示する。

## 曲情報がない場合

```text
曲情報                         ジャケット情報
曲情報がありません。           読み込み中…

[ 曲情報を更新 ]
```

## DB読み込み中

```text
曲情報                         ジャケット情報
読み込み中…                    読み込み中…

[ 曲情報を更新 ]  ※無効
```

## ジャケット情報初期作成中

```text
曲情報                         ジャケット情報
更新日時: 2026/07/19 10:00:00  初期情報を作成中…
曲: 1181曲
譜面: 9594譜面

[ 曲情報を更新 ]  ※無効
```

## 曲情報更新中

```text
曲情報                         ジャケット情報
曲情報を更新しています…        version: 1.0
                               収集状況: 120 / 1181
                               レビュー中: 10件

[ 曲情報を更新 ]  ※無効
```

## 通常状態

```text
曲情報                         ジャケット情報
更新日時: 2026/07/19 10:00:00  version: 1.0
曲: 1181曲                     収集状況: 120 / 1181
譜面: 9594譜面                 レビュー中: 10件

[ 曲情報を更新 ]
```

## 更新失敗時

既存DBが利用可能なら、更新失敗によって現在の情報を失わない。

```text
曲情報                         ジャケット情報
更新に失敗しました。            version: 1.0
前回更新: 2026/07/19 10:00:00   収集状況: 120 / 1181
曲: 1181曲                     レビュー中: 10件
譜面: 9594譜面

[ 曲情報を更新 ]

エラー: <短い理由>
```

詳細なstack traceは通常画面へ表示しない。

## 処理の競合

曲情報またはジャケット情報に対する初期化・更新処理中は、同種の処理を重ねて開始しない。

更新処理と収集処理が同時実行できない場合は、以下のどちらかへ統一する。

- 更新開始時に収集を停止する。
- 収集中は更新ボタンを無効化する。

実装前に既存のDB接続・収集処理境界を確認し、安全で単純な方を採用する。

# ジャケット情報の初期作成

曲情報の読み込みに成功し、ジャケット情報DBが存在しない場合に自動作成する。

最低限、以下を作成する。

- catalog schema
- catalog metadata
- Master上のGP対象曲を未収集として扱える初期状態

全曲分のrowを事前作成するか、曲情報とのjoinで未収集を導出するかは、既存schemaと実装を確認して決める。画面上の結果は同じにする。

- GP対象曲が収集状況へ表示される。
- 初期状態は未収集になる。
- orphanや未割当を生成しない。

# 収集状況

## 集計表示

現在の内部用語を日本語へ変更する。

```text
収集済み: 0 / レビュー待ち: 0 / 未収集: 1181 / 曲未特定: 0
曲情報に存在しないデータ: 0 / 未割当: 0
```

## 状態ラベル

| 内部値 | 表示 |
|---|---|
| `all` | すべて |
| `referenced` | 収集済み |
| `needs_review` | レビュー待ち |
| `uncollected` | 未収集 |
| `unresolved` | 曲未特定 |
| `orphan` | 曲情報に存在しないデータ |

`未割当 unresolved`のように内部分類を連結した表示は避け、人間向けの独立したラベルを定義する。

## 理由ラベル

理由プルダウンと理由列も、内部値をそのまま表示しない。

実装時に実在するreason値を列挙し、以下の形式で表示マッピングを定義する。

| 内部値 | 表示例 |
|---|---|
| capture mismatch系 | 取得画面が一致しない |
| OCR unresolved系 | 曲名を特定できない |
| duplicate系 | 重複候補 |
| missing image系 | 画像が見つからない |
| stale / drift系 | 元データが更新されている |

未知のreason値は隠さず、`不明: <内部値>`として表示する。

## 一覧列

基本列は以下とする。

```text
状態 / 曲名 / アーティスト / 登録数 / 理由 / song ID
```

- `song ID`は診断に有用なため残してよい。
- `登録数`が何を数えるかは、既存実装を確認して表示名を具体化する。
  - ジャケット画像数なら`登録画像数`
  - reference数なら`登録数`
- 状態・理由は日本語表示にする。

# 実装前に確定する事項

次Goalへ昇格する際に、以下をコードと実データから確認する。

1. **更新日時の定義**
   - DBファイル更新日時ではなく、曲情報の正常更新完了日時を表示する。
   - 既存metadataに時刻がなければ保存方法を決める。

2. **曲情報更新元と失敗時動作**
   - 更新元、ネットワーク失敗、入力不正時の扱いを確認する。
   - 更新は一時ファイルへ作成し、成功後に置換することが望ましい。

3. **曲情報更新時のジャケット情報追従**
   - 新曲追加時に未収集として表示されること。
   - 削除・GP対象外化された曲をどう扱うか。
   - 既存referenceを失わないこと。

4. **収集数の定義**
   - `120 / 1181`の分子を、収集済み曲数とするかreference登録済み曲数とするか。
   - review待ちを収集済みに含めるか。

5. **レビュー中件数の定義**
   - `needs_review`のみか、manual review画面の`unreviewed`と`hold`を含むか。
   - 画面間で同じ定義を使う。

6. **処理日時とversion**
   - catalog versionの表示形式を`1`、`v1`、`1.0`のどれにするか。
   - 既存schemaの値を事実どおり表示し、UIだけで架空の小数versionへ変換しない。

7. **更新と収集の排他関係**
   - 曲情報更新中に収集を継続できるか。
   - 継続できない場合の停止・再開動作。

8. **キャンセル操作**
   - 現在の`処理を取消`が必要か。
   - 安全に中断できない処理ならボタンを置かず、完了を待つ。

# 非要件

- 任意のmaster/catalogファイルを選択する通常UI
- 詳細なDB管理画面
- DB hashの通常表示
- migration履歴viewer
- stack traceの通常表示
- 複数catalogの切り替え
- 一般ユーザー向けの保存場所カスタマイズ

# Goal候補の完了条件

- UI上の`master` / `catalog`表記が`曲情報` / `ジャケット情報`へ置き換わる。
- hashではなく曲情報の正常更新日時を表示する。
- DBファイルを手動選択せず起動・初期化できる。
- 曲情報がない状態から画面操作で更新できる。
- ジャケット情報DBがない場合に自動作成される。
- 読み込み・更新・初期作成・通常・失敗状態が区別して表示される。
- 排他処理中は更新ボタンが無効になる。
- 収集状況と理由が日本語ラベルで表示される。
- 新曲追加後も既存ジャケット情報を失わず、未収集曲として追従する。
- 対象test、Ruff、構文検査、`git diff --check`が成功する。
