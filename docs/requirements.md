# DanceDanceRevolution GRAND PRIX スコア保存機能 要求定義

## 1. 概要

DanceDanceRevolution GRAND PRIX のプレー結果を、Windows上で自動取得し、ローカルDBへ保存する。

対象はWindowsのみとする。DanceDanceRevolution GRAND PRIX がWindows対応のみであること、および画面キャプチャ・常駐アプリ・OCR処理をWindows前提で設計するため。

本プロジェクトは大きく以下の2機能に分ける。

- マスタ取得機能
- スコア情報取得/保存機能

## 2. 全体方針

### 2.1 マスタ取得機能

- GitHub Actions + Python で実装する。
- BEMANIWiki の DanceDanceRevolution GRAND PRIX 全曲リストから楽曲・譜面情報を取得する。
- 取得元URL:
  - https://bemaniwiki.com/?DanceDanceRevolution+GRAND+PRIX/%E5%85%A8%E6%9B%B2%E3%83%AA%E3%82%B9%E3%83%88
- HTMLテーブルを解析し、SQLite形式のマスタDBを生成する。
- 生成したマスタDBは GitHub Releases の成果物として配布する。
- ローカルの個人スコアDBとは分離する。

### 2.2 スコア情報取得/保存機能

- C# / .NET 10 LTS のWindows常駐デスクトップアプリとして実装する。
- DDR GRAND PRIX のゲームウィンドウを直接キャプチャする。
- OBS経由やゲーム内部通信への介入は初期対象外とする。
- リザルト表示フェーズを自動検出し、画面情報からスコア情報を抽出する。
- 取得結果の確信度が十分な場合のみ、ローカルのスコアDBへ完全自動保存する。
- 低確信度・不一致・解析失敗の場合はDB保存せず、画像とログを保存する。

## 3. 対象画面と入力元

DanceDanceRevolution GRAND PRIX は主に以下のフェーズに分かれる。

- 選曲フェーズ
- プレーフェーズ
- リザルト表示フェーズ

本機能はリザルト表示フェーズのみを保存対象とする。

入力元は DDR GRAND PRIX のWindowsウィンドウを直接キャプチャした画像とする。

画面取得方式は Windows Graphics Capture API を第一候補とする。取得対象ウィンドウの特定、キャプチャの安定性、Windows常駐アプリとの相性を重視する。

## 4. マスタDB要件

### 4.1 取得項目

BEMANIWiki の全曲リストから、少なくとも以下を取得する。

- 曲名
- アーティスト名
- バージョン、初出、分類
- BPM
- プレースタイル
  - SINGLE
  - DOUBLE
- 難易度
  - BEGINNER
  - BASIC
  - DIFFICULT
  - EXPERT
  - CHALLENGE
- レベル
- ショックアロー等の注記
- 削除曲、限定曲、パック関連など判別可能な注記

### 4.2 DB構成

マスタDBは `ddrgp-master.sqlite` とする。

主なテーブルは以下を想定する。

- `songs`
  - 楽曲単位の情報を保持する。
- `charts`
  - 楽曲ごとのSP/DP・難易度・レベルを保持する。
- `master_metadata`
  - 生成日時、取得元URL、取得元ページ更新情報、生成スクリプトバージョンを保持する。
- `source_snapshots`
  - 解析元HTMLのハッシュ、取得日時、必要に応じた元データ参照情報を保持する。

### 4.3 更新・配布

- GitHub Actions で定期実行する。
- 手動実行も可能にする。
- 成果物は GitHub Releases に添付する。
- スコア取得アプリは起動時または手動操作でマスタDB更新を確認する。
- 個人スコアDBはマスタDB更新時に上書きしない。

## 5. スコアDB要件

### 5.1 保存方針

- 全プレー履歴を保存する。
- 同一譜面の自己ベストのみ保存する方式にはしない。
- ベストスコアや統計は、保存済み履歴からビューまたはクエリで算出する。

### 5.2 保存項目

ローカルスコアDBは `ddrgp-scores.sqlite` とする。

`plays` テーブルに1プレー1レコードで保存する。

主な保存項目は以下とする。

- プレー日時
- マスタDBバージョン
- 曲ID
- 譜面ID
- 曲名OCR結果
- プレースタイル
  - SINGLE
  - DOUBLE
- 難易度
- レベル
- スコア
- ランク
- クリア種別
- 判定数
  - MARVELOUS
  - PERFECT
  - GREAT
  - GOOD
  - MISS
  - OK
- 最大コンボ
- キャプチャ画像ハッシュ
- OCR/解析確信度
- アプリバージョン

### 5.3 重複保存防止

同じリザルト画面が数秒間表示され続けるため、重複保存防止を行う。

以下を組み合わせて同一リザルトを判定する。

- キャプチャ画像ハッシュ
- 曲ID
- 譜面ID
- スコア
- 判定数
- 短時間内の同一結果

同一リザルトと判定した場合は追加保存しない。

## 6. 画面解析要件

### 6.1 解析方式

初期方針は固定ROI + OCR とする。

- 基準解像度へ正規化する。
- リザルト画面上の固定領域を切り出す。
- OpenCvSharp で二値化、ノイズ除去、拡大、コントラスト調整などの前処理を行う。
- Tesseract OCR の `jpn+eng` を利用する。
- スコアや判定数など数字中心の領域は、数字向けに前処理・文字制限を行う。
- ランクや難易度は、OCRに加えてテンプレート照合の併用を検討する。

### 6.2 解像度・表示倍率対応

- 初期版から自動補正 + 設定画面でのROI微調整に対応する。
- フルHD固定など特定環境限定にはしない。
- ただし、精度保証対象となる解像度・ウィンドウサイズは実機スクリーンショット収集後に定義する。

### 6.3 マスタ照合

OCRで取得した曲名、プレースタイル、難易度をマスタDBと照合する。

曲名は表記揺れやOCR誤りを考慮し、ファジーマッチを行う。

保存条件は以下とする。

- リザルト画面であることを検出できる。
- 曲を一意に特定できる。
- 譜面を一意に特定できる。
- スコア・判定数など必須項目を読み取れる。
- 総合確信度が保存閾値以上である。

条件を満たさない場合はDB保存しない。

## 7. 失敗時の扱い

低確信度、解析失敗、マスタ照合失敗、重複疑いなどの場合はスコアDBへ保存しない。

代わりに以下を保存する。

- 失敗時キャプチャ画像
- JSON形式の解析ログ
- 失敗理由
- OCR生テキスト
- 候補曲一覧
- 各項目の確信度

保存先はローカルアプリデータ配下の `failed-captures/` と `logs/` を想定する。

## 8. 常駐アプリ要件

### 8.1 基本動作

- Windows起動後、ユーザー操作でアプリを起動する。
- タスクトレイ常駐を基本とする。
- DDR GRAND PRIX 起動中は対象ウィンドウを検出する。
- リザルト画面検出時に自動解析・自動保存する。
- 完全自動保存を基本とし、保存前確認ダイアログは出さない。

### 8.2 画面・設定

最低限、以下の画面を用意する。

- 現在の監視状態
- 最新保存結果
- マスタDB更新状態
- ROI調整画面
- 失敗ログ一覧
- 保存済み履歴の簡易一覧

初期UIは保存済み履歴の簡易一覧を最初の縦断価値として先行実装する。正式個人スコアDB version 1と生成済みマスタDBをread-onlyで開き、曲・譜面情報つきプレー履歴、プレー詳細、保存済み履歴から算出した譜面別自己ベストを表示する。監視状態、実キャプチャ、ROI調整、失敗ログ一覧、マスタDB更新は後続段階で接続する。

### 8.3 ログ

通常ログと解析失敗ログを分ける。

ログには以下を含める。

- アプリ起動・終了
- 対象ウィンドウ検出状態
- マスタDBバージョン
- リザルト検出
- 保存成功
- 保存スキップ理由
- 例外情報

## 9. 初期開発時の検証素材

初期開発では実際に DanceDanceRevolution GRAND PRIX を起動し、リザルト画面のスクリーンショットを収集する。

収集対象は以下とする。

- SINGLE / DOUBLE
- 複数難易度
- 複数ランク
- 複数スコア桁
- クリア/未クリア系表示差分
- フルHD
- ウィンドウ表示
- Windows表示倍率違い
- 曲名が英字、日本語、記号を含むケース
- 同名・類似名がありそうな曲

収集した画像は自動テスト用fixtureとして利用する。

## 10. テスト要件

### 10.1 マスタ取得テスト

- HTML取得成功時にSQLiteが生成されること。
- BEMANIWikiのテーブル構造から曲・譜面情報を正しく抽出できること。
- セル結合、注記、削除曲、限定曲、CHALLENGEなし、SP/DP片方のみのケースを扱えること。
- 取得元HTML構造が変化した場合にエラーを検出できること。

### 10.2 画像解析テスト

- 実機スクリーンショットからリザルト画面を検出できること。
- 必須項目をOCRできること。
- マスタDBと一意に照合できること。
- 低確信度時にDB保存しないこと。
- 失敗時に画像とログを残すこと。

### 10.3 DB保存テスト

- 1プレー1レコードで保存されること。
- 同一リザルトの重複保存を防止できること。
- マスタDB更新後も過去履歴が参照できること。
- 個人スコアDBがマスタDB更新で上書きされないこと。

## 11. 受け入れ基準

初期版の受け入れ基準は以下とする。

- 実機スクリーンショットの検証セットに対して、自動保存成功率95%以上を目標とする。
- 誤保存は0件を目標とする。
- 低確信度ケースは保存せず、必ずログと画像が残る。
- マスタDBとスコアDBが分離されている。
- 同一リザルトを複数回保存しない。
- GitHub ActionsでマスタDBを生成できる。

## 12. 初期対象外

以下は初期対象外とする。

- OBS WebSocket経由の画面取得
- DDR GRAND PRIX の内部通信解析
- ゲームプロセスへのメモリ読み取り・改変
- クラウド同期
- スマートフォン対応
- macOS / Linux 対応
- 保存前確認を前提とした半自動運用
- 公式サービスへの自動送信

## 13. 未決事項

実装前または検証時に決める事項は以下とする。

- 精度保証対象の解像度・ウィンドウサイズ
- ROI初期値
- OCR確信度の保存閾値
- 曲名ファジーマッチの閾値
- ローカルデータ保存先の最終パス
- 配布形態
  - zip
  - installer
  - self-contained exe
- .NET UI技術の最終選定
  - WPFを第一候補とする
  - WinUI 3は必要性が出た場合に再検討する
